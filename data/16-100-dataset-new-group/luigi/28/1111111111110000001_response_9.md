The bug in the `table_exists` function is that when the `partition` parameter is not None, it invokes `self.partition_spec(partition)` without a proper database specified. This sometimes results in the `run_hive_cmd` function returning an empty string, leading to false negatives.

The fix for this bug is to provide the database for the `self.partition_spec` function call to ensure that the command executed in the `run_hive_cmd` will return the expected results.

Here's the corrected version of the `table_exists` function:

```python
def table_exists(self, table, database='default', partition=None):
    if partition is None:
        stdout = run_hive_cmd('use {0}; show tables like "{1}";'.format(database, table))

        return stdout and table in stdout
    else:
        partition_string = " and ".join(["{0}='{1}'".format(k, v) for k,v in partition.items()])
        command = "use {0}; show partitions {1} partition ({2})".format(database, table, partition_string)
        stdout = run_hive_cmd(command)

        if stdout:
            return True
        else:
            return False
```

In this corrected version, the `partition_spec` function is removed, and instead, the partition lookup command is constructed directly in the `table_exists` function with the proper database specified. This should resolve the issue where the `run_hive_cmd` function sometimes returned an empty string, causing false negatives.
