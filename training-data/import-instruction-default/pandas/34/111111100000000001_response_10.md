The potential error location within the problematic function is the calculation of `last` while calling the `_get_timestamp_range_edges` function.

The bug's cause is that the `last` value is being calculated using the `replace()` function, which is causing the nanosecond part to be swallowed. This leads to the last bin possibly being slightly before the end, resulting in a `Values falls after last bin` error.
